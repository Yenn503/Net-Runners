## 🔍 ما هو Net-Runner؟

Net-Runner هو إطار عمل متعدد الوكلاء لاختبارات الأمان، ومصمم للعمل باللغة الطبيعية.

أنت تربط نموذج LLM، وتصف الهدف والمطلوب بلغة عادية، ثم يقوم Net-Runner بتشغيل المحرك الكامل:

- يكتشف نية التقييم
- ينشئ بيئة تشغيل `.netrunner/` داخل المشروع
- يحقن النطاق وسياق مسار العمل داخل الجلسة
- يوجّه العمل إلى الوكلاء المتخصصين عند الحاجة
- يسجّل الأدلة والذاكرة والتقارير أثناء تقدم التقييم

```text
أنت تصف المهمة.
Net-Runner يخطط، ويفوض، وينفذ، ويتذكر، ويصدر التقارير.
```

---

## ✨ لماذا يستخدمه الناس

- **اللغة الطبيعية أولًا** — لا حاجة لحفظ أوامر قبل البدء
- **نظام واحد متكامل** — الوكلاء والأدوات والأدلة والذاكرة والتقارير تعمل في نفس التدفق
- **وكلاء متخصصون** — أدوار الاستطلاع والويب وواجهات API والشبكات والاستغلال وAD وإعادة الاختبار والأدلة والتقارير جاهزة مسبقًا
- **ذاكرة مستمرة** — يمكن استرجاع السياق المفيد عبر الجلسات باسترجاع مبني على RAG
- **الأدلة أولًا** — الاكتشافات وخطوات التنفيذ والموافقات والتقارير تبقى مرتبطة بنفس المهمة

## 🚀 ابدأ من هنا

### 1. التثبيت والبناء

```bash
bun install
bun run build
```

### 2. ربط النموذج

#### `ANTHROPIC_API_KEY`

```bash
export ANTHROPIC_API_KEY="sk-ant-..."
node dist/cli.mjs
```

#### OpenAI

```bash
export OPENAI_API_KEY="sk-..."
export OPENAI_MODEL="gpt-4o"
node dist/cli.mjs
```

#### Google Gemini

```bash
export GEMINI_API_KEY="AIza..."
export GEMINI_MODEL="gemini-2.5-pro"
node dist/cli.mjs
```

#### Ollama

```bash
ollama serve
ollama pull llama3.1:8b
export OPENAI_BASE_URL="http://localhost:11434/v1"
export OPENAI_MODEL="llama3.1:8b"
```

### 3. تحدث بشكل طبيعي

```text
قيّم https://target.example. ابدأ بالاستطلاع، وحدد سطح الهجوم الأساسي، ثم تحقق من أكثر المشكلات احتمالًا واحفظ الأدلة أثناء التقدم.
```

سيقوم Net-Runner بتحديد الهدف، وبدء المهمة، وحقن السياق الصحيح، ثم يبدأ باستخدام وقت التشغيل الوكيلي الخاص به.

## ⚙️ كيف يعمل

```text
أنت
  ↓
جلسة LLM الرئيسية
  ↓
سياق تشغيل Net-Runner
  ↓
وكلاء متخصصون + أدوات + ذاكرة + أدلة
  ↓
مخرجات تقييم منظمة
```

| الخطوة | ماذا يفعل Net-Runner |
|--------|----------------------|
| **1. الكشف** | يتعرف على نية التقييم ونوع الهدف ومسار العمل الأرجح |
| **2. التهيئة** | ينشئ حالة `.netrunner/` للمهمة إذا لم تكن موجودة بعد |
| **3. الحقن** | يضيف النطاق وحدود التأثير ومسار العمل والمهارات الافتراضية إلى الجلسة الحية |
| **4. التوجيه** | يستخدم وقت التشغيل الرئيسي والوكلاء المتخصصين معًا بدلًا من إجبارك على إدارة الأوامر يدويًا |
| **5. الحماية** | يطبق حواجز داخلية على الأفعال التدميرية أو المستمرة أو الخارجة عن النطاق |
| **6. التسجيل** | يحفظ الأدلة وخطوات التنفيذ والاكتشافات والمراجعات والذاكرة والتقارير في نفس البيئة |

في المسار الطبيعي يمكنك:

- إعطاؤه هدفًا
- إخباره بنوع التقييم المطلوب
- طلب الاستمرار أو التعمق أو إعادة الاختبار أو التلخيص أو التقرير
- تركه يستخدم البيئة والأدوات والذاكرة والوكلاء المتاحين لديه بالفعل

## 🕵️ الوكلاء

يحتفظ Net-Runner بالتدفق الوكيلي العام الأصلي، ويضيف فوقه أدوارًا أمنية متخصصة.

| الوكيل | ماذا يفعل |
|:-------|:----------|
| **Engagement Lead** | ينسق التقييم، ويختار مراحل مسار العمل، ويوجه العمل |
| **Recon Specialist** | يحدد المضيفين والخدمات والنطاقات الفرعية والتقنيات وسطح الهجوم |
| **Web Testing Specialist** | يختبر المسارات والمعاملات وتدفقات المصادقة وثغرات الويب |
| **API Testing Specialist** | يختبر واجهات API والمخططات وJWT ومسارات IDOR وانتقالات الحالة |
| **Network Testing Specialist** | يعالج تعداد الخدمات والتحقق الشبكي والاختبارات على مستوى المضيف |
| **Exploit Specialist** | يتحقق من الأثر الفعلي بشكل مضبوط |
| **Privilege Escalation Specialist** | يعالج مسارات التصعيد بعد الوصول الأولي |
| **Lateral Movement Specialist** | يعالج التمحور ومسارات الثقة والحركة بين عدة مضيفين |
| **AD Specialist** | يركز على Active Directory وKerberos |
| **Retest Specialist** | يعيد إنتاج النتائج ويتحقق من الإصلاحات |
| **Evidence Specialist** | ينظم القطع الأثرية والأدلة القابلة للتتبع |
| **Reporting Specialist** | يحول الأدلة إلى تقرير واضح |

ولا تزال وكلاء وقت التشغيل الأساسية مثل `general-purpose` و`Explore` و`Plan` و`verification` جزءًا من النظام.

## 🧱 بنية المشروع

```text
.netrunner/
├── engagement.json
├── run-state.json
├── evidence/
│   └── ledger.jsonl
├── findings/
├── reports/
├── artifacts/
├── memory/
│   ├── private.md
│   ├── team.md
│   └── agents/
└── instructions/
```

- `engagement.json` — مسار العمل الحالي، والأهداف، وحد التأثير، والقيود
- `run-state.json` — خطوات التنفيذ والمراجعات المعلقة
- `evidence/` — سجل أدلة يضاف إليه فقط
- `findings/` — مخرجات منظمة للنتائج
- `reports/` — التقارير التي تم توليدها
- `artifacts/` — المخرجات المجمعة والملفات الداعمة
- `memory/` — ذاكرة مستمرة للمشغل والفريق والوكلاء
- `instructions/` — تعليمات تشغيل خاصة بالمشروع
